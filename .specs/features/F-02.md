## Resumo

Modelo de usuario customizado com login por email e fluxo de verificacao por codigo de 6 digitos. Fundacao de autenticacao para todo o sistema.

## Contexto

O Django exige que o User customizado seja definido antes do primeiro `makemigrations`. A plataforma usa email como identificador principal (decisao STATE.md 2026-04-11). O fluxo de verificacao garante que apenas emails validos acessem o sistema.

## Requisitos Funcionais

- [x] RF-01: User customizado com email como campo de login (`USERNAME_FIELD`)
- [x] RF-02: Username mantido como campo de display, mas login via email
- [x] RF-03: Manager customizado (`UserManager`) que normaliza email na criacao
- [x] RF-04: Modelo `EmailVerification` com codigo de 6 digitos e expiracao configuravel
- [x] RF-05: Verificacao expira em 15 minutos por padrao
- [x] RF-06: Codigo so pode ser usado uma vez (`mark_used`)

## Regras de Negocio

- RN-01: Email deve ser unico no sistema (constraint de banco)
- RN-02: Superuser obrigatoriamente deve ter `is_staff=True` e `is_superuser=True`
- RN-03: Codigo de verificacao invalido se ja usado (`used_at != None`) ou se expirou
- RN-04: Cada verificacao pertence a um usuario (FK com cascade delete)

## Design Tecnico

### Entidades / Models

| Model             | Campos principais                                      | Notas                                     |
|-------------------|--------------------------------------------------------|-------------------------------------------|
| User              | email (unique), username, herda AbstractUser           | `USERNAME_FIELD = "email"`, manager custom |
| EmailVerification | user (FK), code (6 chars), created_at, expires_at, used_at | Ordering por `-created_at`            |

### Fluxos Principais

1. **Criacao de usuario**: `UserManager.create_user(email, username, password)` â†’ normaliza email â†’ salva
2. **Verificacao de email**: cria `EmailVerification` â†’ gera codigo 6 digitos â†’ `is_valid()` checa expiracao e uso â†’ `mark_used()` consome o codigo

## Dependencias

- F-01 (setup do projeto, estrutura modular)
- Nenhum pacote externo adicional

## Criterios de Aceitacao

1. `User` usa email como `USERNAME_FIELD`
2. `create_user` e `create_superuser` funcionam corretamente com validacoes
3. Email duplicado causa `IntegrityError`
4. `EmailVerification.is_valid()` retorna `False` para codigos expirados ou ja usados
5. `mark_used()` registra timestamp e invalida o codigo
6. Codigos gerados tem exatamente 6 digitos, zero-padded

## Decisoes

| Decisao                           | Alternativas consideradas       | Justificativa                                                    |
|-----------------------------------|---------------------------------|------------------------------------------------------------------|
| AbstractUser (nao AbstractBaseUser)| AbstractBaseUser                | Minima mudanca, mantendo admin/permissoes nativos                |
| Email como USERNAME_FIELD          | Username padrao                 | Requisito de negocio â€” login por email                           |
| Codigo 6 digitos via `secrets`     | UUID, token longo               | UX mais simples para verificacao por email                       |
| Expiracao em campo `expires_at`    | Calculo dinamico no `is_valid`  | Permite customizar expiracao por registro, query mais eficiente  |
