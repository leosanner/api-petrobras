# F-03 — Autenticacao e Autorizacao (Session Cookies)

## Resumo

Sistema completo de autenticacao via session cookies com registro, verificacao de email, login/logout, reset e troca de senha. Configura CORS para comunicacao cross-origin com frontend React.

## Contexto

A F-02 entregou o modelo User customizado (email login) e EmailVerification (codigo 6 digitos). Decisoes de arquitetura (STATE.md 2026-04-11): session cookies sem JWT (~500 users, revogacao importa), login por email+senha, self-registration aberta, Resend como email provider, papeis via Group+Permission nativos do Django.

## Requisitos Funcionais

- [ ] RF-01: Endpoint de registro cria user inativo e envia email com codigo de verificacao
- [ ] RF-02: Endpoint de verificacao valida codigo e ativa user
- [ ] RF-03: Endpoint de login autentica por email+senha e retorna session cookie
- [ ] RF-04: Endpoint de logout encerra sessao
- [ ] RF-05: Endpoint de password reset envia codigo por email (sem enumeracao de emails)
- [ ] RF-06: Endpoint de password reset confirm valida codigo e seta nova senha
- [ ] RF-07: Endpoint de password change permite user autenticado trocar senha
- [ ] RF-08: Endpoint /me/ retorna info do user autenticado
- [ ] RF-09: Endpoint CSRF permite SPA obter cookie csrftoken
- [ ] RF-10: CORS configurado para permitir cookies cross-origin (Cloudflare Pages → Render)
- [ ] RF-11: Envio de email via Resend Python SDK

## Regras de Negocio

- RN-01: User e criado com `is_active=False` ate verificar email
- RN-02: Login rejeitado para users com `is_active=False`
- RN-03: Password reset sempre retorna 200, independente de o email existir (previne enumeracao)
- RN-04: Senhas devem passar pelos validators do Django (similaridade, minimo 8 chars, comuns, numericas)
- RN-05: Session cookie: HttpOnly, Secure (prod), SameSite=None (prod) / Lax (dev)
- RN-06: CSRF cookie: nao HttpOnly (frontend JS precisa ler), Secure (prod), SameSite=None (prod) / Lax (dev)
- RN-07: Session expira em 7 dias
- RN-08: Codigos de verificacao sao single-use e expiram (15min email, 30min password reset)

## Design Tecnico

### Entidades / Models

| Model | Campos principais | Notas |
|-------|-------------------|-------|
| BaseVerificationCode | user (FK), code (6 chars), created_at, expires_at, used_at | Abstract. Template method: EXPIRATION_MINUTES, is_valid(), mark_used() |
| EmailVerification | herda BaseVerificationCode | EXPIRATION_MINUTES=15. Tabela existente (accounts_emailverification) |
| PasswordResetCode | herda BaseVerificationCode | EXPIRATION_MINUTES=30. Tabela nova (accounts_passwordresetcode) |

### Endpoints

| Metodo | Rota | Permissao | Descricao |
|--------|------|-----------|-----------|
| POST | `/api/auth/register/` | AllowAny | Cria user inativo, envia codigo |
| POST | `/api/auth/verify-email/` | AllowAny | Valida codigo, ativa user |
| POST | `/api/auth/login/` | AllowAny | Autentica, cria sessao |
| POST | `/api/auth/logout/` | IsAuthenticated | Encerra sessao |
| POST | `/api/auth/password-reset/` | AllowAny | Envia codigo de reset |
| POST | `/api/auth/password-reset-confirm/` | AllowAny | Valida codigo + nova senha |
| POST | `/api/auth/password-change/` | IsAuthenticated | Troca senha |
| GET | `/api/auth/me/` | IsAuthenticated | Info do user |
| GET | `/api/auth/csrf/` | AllowAny | Seta cookie CSRF |

### Fluxos Principais

1. **Registro**: POST /register/ → cria User(is_active=False) → cria EmailVerification → envia email via Resend → 201
2. **Verificacao**: POST /verify-email/ (email, code) → busca codigo valido → mark_used() → user.is_active=True → 200
3. **Login**: POST /login/ (email, password) → authenticate() → login(request, user) → Set-Cookie: sessionid → 200
4. **Logout**: POST /logout/ → logout(request) → sessao encerrada → 200
5. **Password reset**: POST /password-reset/ (email) → cria PasswordResetCode → envia email → 200 (sempre)
6. **Password reset confirm**: POST /password-reset-confirm/ (email, code, new_password) → valida codigo → set_password() → mark_used() → 200
7. **Password change**: POST /password-change/ (current_password, new_password) → valida senha atual → set_password() → 200

## Dependencias

- F-02 (User model, EmailVerification)
- `django-cors-headers` (CORS cross-origin)
- `resend` (SDK Python para envio de email)

## Criterios de Aceitacao

1. POST /register/ cria user com is_active=False e EmailVerification com codigo
2. POST /verify-email/ com codigo valido ativa user (is_active=True)
3. POST /verify-email/ com codigo invalido/expirado retorna 400
4. POST /login/ com credenciais validas retorna session cookie
5. POST /login/ com user inativo retorna 400
6. POST /logout/ encerra sessao (sessionid cookie invalidado)
7. GET /me/ autenticado retorna dados do user
8. GET /me/ nao autenticado retorna 403
9. POST /password-reset/ com email existente envia codigo; com email inexistente retorna 200 sem envio
10. POST /password-reset-confirm/ com codigo valido troca senha
11. POST /password-change/ com senha atual correta troca senha
12. GET /csrf/ retorna cookie csrftoken
13. BaseVerificationCode e abstract; EmailVerification e PasswordResetCode herdam com EXPIRATION_MINUTES diferentes
14. Emails enviados via Resend SDK

## Decisoes

| Decisao | Alternativas consideradas | Justificativa |
|---------|---------------------------|---------------|
| Session cookies (sem JWT) | JWT, OAuth | ~500 users, revogacao nativa, Django entrega pronto |
| Template Method (BaseVerificationCode abstract) | Modelo unico com campo purpose; modelos separados sem heranca | Reutiliza logica comum, cada subclasse especializa expiracao e comportamento |
| Resend Python SDK | SMTP relay com send_mail() | API moderna, chamada direta no service layer |
| Endpoint /csrf/ dedicado | Frontend obtem CSRF via outra request | Padrao claro para SPA, explicito e previsivel |
| CSRF_COOKIE_HTTPONLY=False | True | Frontend JS precisa ler cookie para enviar header X-CSRFToken |
| SameSite=None + Secure (prod) | Lax | Frontend em dominio diferente (Cloudflare Pages) requer cross-origin cookies |
